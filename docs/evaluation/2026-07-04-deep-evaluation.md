## 1. Goal Alignment Audit

### G1 — The blurred boundary between player, agent, and world

**Implemented.** There is exactly one real mechanical substrate for the blur, and it is genuinely good: a single trace ledger. Human speech (`PLAYER_SPEAK`, server/handlers.py:493), human chat (`PLAYER_CHAT`, handlers.py:594), and agent traversal events (bus recorder, handlers.py:82–86) all land in one `world_mutations` table and all re-enter play through the same `_history_block` in the consciousness prompt (consciousness/__init__.py:349–358). A node cannot tell — and does not report — whether its visitors were human or agent beyond a name string. CLI players are even recorded as `player_name=None` and voiced as "an unknown presence" (consciousness/__init__.py:354), which is accidentally on-theme. At the transport layer, `chat`, `causal_event`, `agent_encounter`, and `puzzle_solved` all arrive on one WebSocket surface, so a spectator can't type-check the actor.

**Partial.** The CLI's unrecognized-input-falls-through-to-speak (interface/__init__.py:224–225) is a genuine "everything is addressable" gesture — but keyless it means every typo produces an authentication error, the opposite of blur.

**Missing.** The blur has no speech-side mechanism at all. Node voice (`speak()`) and agent voice (`voice_agent()`) are separate endpoints with separate system-prompt "bibles"; nothing ever routes a `/speak` to an agent, no agent settles into a node, and neither nodes nor agents can ever initiate contact — every utterance in the game is client-initiated request/response. `/agent/voice` doesn't even fetch node history or agent memory (handlers.py:707–710): the "agent" you converse with is a stateless improvisation with zero connection to the FSM agent whose traces are in the room. There are no Claude calls anywhere in `agents/` — agent behavior is a 44-line threshold FSM (agents/behaviors.py). The README's "deliberately open question" of whether you're talking to a node or an agent who settled into it is answered unambiguously by the routing table on every call.

**Divergent.** Two things actively undo the blur. First, `/speak` constructs the node from client-supplied body fields — name, level, properties — with no check against the generated world (handlers.py:468–487), and records persistent mutations under that client-supplied name. The boundary isn't blurred; it's *forgeable*, by clients only, never by the world. Second, both UIs taxonomize relentlessly: "Explorers" roster, "Watch Agent Traverse" button, "Speak to Room · Vault-3" labels. The presentation layer un-blurs what the ledger blurred.

### G2 — The multiverse is always running and persistent

**Implemented.** The trace overlay is real and well-built. SQLite persists mutations, per-node `ripple_score` (written through on every bus fire, handlers.py:89–98; rehydrated on every request, handlers.py:59–61), agent visited-sets, puzzle results, the image cache, invite keys, and cross-device resume positions (`/position`, migration 0005). Verified hands-on: CLI activity appeared in the server's `/worlds`; a session accreted 31 mutations visible in `/history`; a second browser with the same invite key resumed at the same node. Traces genuinely outlive processes.

**Partial.** But what persists is an annotation layer, not a world. The tree is regenerated from seed on every endpoint call (handlers.py:57, 378, 748, 826, 855); no table stores node properties and no code path ever mutates `node.properties` after generation. "The world evolves" reduces to: an append-only event log plus one float per node. Meanwhile rooms, presence, co-op puzzle sessions, and rate-limit state are per-process RAM (server/rooms.py) — `fly.toml:27–28` pins an always-on VM explicitly to babysit them, and every deploy silently wipes them.

**Missing.** Nothing runs unattended. Anywhere. The only spawned thread is the shutdown handler (server/__init__.py:38); the only production `while True` loops are the WS receive pump (handlers.py:567) and the CLI REPL (interface/__init__.py:168). Verified by grep and by execution: with zero players connected, zero state changes. Every causal event, agent step, and ripple originates inside an HTTP request. "You enter and find it already in motion" is, mechanically, "you enter and trigger all the motion yourself." The background causality worker is deferred to phase-2c (docs/roadmap/phase-2-scale.md:161–164). Relatedly, the world has no recoverable age: `save_world` is `INSERT OR REPLACE` without `created_at` (persistence/__init__.py:165–171), called on every `GET /world` — each visit re-births the world's timestamp.

**Divergent.** Three findings here are worse than "not yet implemented":

1. **Agent memory bricks agents.** `Agent.visited` stores `node.id`, which is `uuid.uuid4()` minted fresh at every regeneration (multiverse/node.py:14). `/agent` saves those UUIDs and restores them into a *newly generated* tree (handlers.py:387–396). Verified by execution (seed 7): run 1 visits 50 nodes; restored memory overlaps the new tree's IDs 0/50; and because `traverse` seeds `visited` from memory and caps on `len(self.visited) >= max_nodes` (agents/agent.py:94, 103), the stale UUIDs consume the entire visit budget — **run 2 visits 0 nodes**. The one mechanism built for agent continuity makes every named agent inert on its second run (at any fixed `max_nodes` ≤ total known).
2. **The same seed is not the same world.** One RNG feeds names, properties, *and* breadth rolls in a depth-first walk (generator.py:180–196), so trees at different depths diverge after the first branch. Verified by execution: seed 42 depth-6 shares **6 of 16** node names with depth-11; seed 7 depth-6 shares 6 of 37. `/world` defaults to depth 6 (handlers.py:53–54) while `/agent` regenerates at full depth 11 (handlers.py:378), and all persistence keys on `(seed, node_name)` — so agent traces mostly land on node names that don't exist in any player's tree, and CLI `play` (also depth 6) writes history into a phantom world. Rooms key on seed alone (handlers.py:557), so players at different depths share a room while inhabiting different trees, receiving "ghost events" that name nodes they can't find. This silently guts G2 and G3 and is untested.
3. **Continuity is disclaimed in writing.** phase-2-scale.md:176: "Persistent game state across cohorts (each beta cohort gets a fresh DB)" is explicitly not planned — cross-cohort continuity of the marketed always-persistent multiverse is a non-goal.

### G3 — Cross-scale causality as connective tissue

**Implemented.** The bus itself is correct and well-tested: `propagate` fires the origin once, cascades bidirectionally with per-hop dampening (default 0.5), halts below 0.05 (causality/__init__.py:83–145). And there are three genuine paths by which one participant's trace reaches another's perception — the strongest connective tissue in the codebase:
1. **Live**: `/observe` traversals and puzzle-solve cascades broadcast every fired event to all sockets in the seed-room (handlers.py:770–787, 920–941); the D3 explorer renders them as strength-scaled ring flashes (explorer.js:538–557). A stranger's solve visibly pulses your screen.
2. **Visual**: persisted `ripple_score` enters `/image`'s style signature and flips the register to "psychedelic" at ≥0.5 (server/imageprompt.py:104; handlers.py:634–646).
3. **Vocal**: mutations surface in `speak()` history (handlers.py:490–491).

**Partial.** Events change nothing but a float. `_fire`'s complete state effect is: log append, `ripple_score += 0.1 × strength` clamped to 1.0, invoke handlers (causality/__init__.py:119–127). All five `EventKind`s cascade identically; `STRUCTURAL_CHANGE` changes no structure; `DANGER_ALERT` is only ever `emit`ted (agents/agent.py:59), so alerts never travel. No event ever touches properties, danger, or children — a destabilized atom cannot cascade into "a volatile region" because volatility is not a mutable quantity. Event strength is also discarded at every persistence boundary: `record_mutation` stores kind + payload only (handlers.py:82–85), so history can't distinguish a 1.0 origin from a 0.05 echo, and the consciousness prompt sees undifferentiated events.

**Missing.** No temporal dimension: cascades are synchronous recursion inside the request — the README's "with dampening **and delay**" has no delay of any kind. `/agent` runs broadcast nothing (no handler on that bus, handlers.py:382–384) — invisible to live players except by polling `/history`. Movement, the most common player act, is never recorded (handlers.py:578–584): being somewhere leaves no causal trace.

**Divergent.** The strengths players see are fabricated. Every display surface recomputes strength as `DAMPENING ** depth` = 0.6^d (handlers.py:772, 932; interface/__init__.py:91–93) while the actual cascade ran at `propagate`'s 0.5 default — and `/observe`'s agent EXPLORE events are `emit`s at flat 1.0 with no cascade at all. Worse, `build_depth_map(target)` maps only the subtree (multiverse/utils.py:20–27), so upward-cascading ancestor events miss the map and broadcast at depth 0 → strength 1.0 regardless of distance. The one place cross-scale attenuation is *visible* — the mechanic the hands-on trace found to be the game's standout beat — displays physics the engine never ran. Separately, `upsert_ripple_score` writes an absolute value computed on the request's private tree (handlers.py:96–97; persistence/__init__.py:308–324): two concurrent players' bumps race, last-writer-wins — structurally, the exact multiplayer scenario G3 describes is where causal contributions silently vanish. And the CLI is a causal dead zone: play-mode solves call nothing (interface/__init__.py:106–110), observe uses an unrecorded local bus — CLI players neither cause nor inherit cascades.

### G4 — Multi-modal interaction; generative-art visual layer responding to world state

**Implemented.** All three modes exist: language (`/speak`, CLI speak, React Speak tab), visual navigation (D3 tree, Pixi scenes, CLI `look`/`map`), ambient observation (`/observe` SSE, CLI ambient mode). And the state→style mapping is real on the server: `derive_modifiers` consumes persisted mutation history and persisted ripple_score with all seven design-doc matrix rows live (server/imageprompt.py:76–106 — ≥5 agent visits→surreal, danger→noir, ≥2 distinct speakers→warm, puzzle→Escher, ≥2 failures→oppressive, corrupted→glitch, ripple≥0.5→psychedelic), and the style signature is folded into the image cache key (handlers.py:642–646) so a modifier flip genuinely forces a fresh render. This is the strongest single G4 artifact in the repo.

**Partial.** "Responding" means "a different static image on a later visit." The client fetches `/image` once per node; a style change surfaces only if a player leaves and returns after the signature flips. The Pixi layer itself is not generative: `renderScene` consumes only `node.level` (one of 8 fixed hex colors), name, child count, and player dots (SceneView.jsx:131–191) — `node.properties` and history drive zero pixels. All generative art is outsourced to fal.ai.

**Missing.** The default client has no visual layer at all: invite URLs land on `/` (pinned by tests/test_frontend_contract.py:202–208), and `explorer.js` never calls `/image` — the headline feature is behind a footnote link ("Visual ↗"). And there is no keyless fallback art: without `FAL_KEY` the server returns an error and `/app` renders a flat void across two-thirds of the viewport (verified by screenshot). The design doc's style drift is *absent*, not degraded, in every keyless deployment. Also: the durable cache stores fal.ai URLs that expire (ADR-002 "Revisit when…") — the world's visual memory rots silently.

**Divergent.** As under G3: every strength number a player sees, in every mode, is a cosmetic function of tree depth rather than engine output.

### G5 — All four interaction patterns under one world model and one set of rules

**Implemented.** Human↔human is the one fully working leg, live and async: WS chat with persistence, join/move/leave presence (verified live at ~1s latency with two browsers), and genuinely pooled co-op puzzle sessions — shared attempt counter, contributor set, first-solver claim under the room lock (rooms.py:137–166), with room-wide `puzzle_solved` + cascade broadcast on solve (handlers.py:907–941).

**Partial.** Agent→human is passive and request-scoped: agents exist for the milliseconds of one traversal; live visibility requires someone to be holding `/observe` open while you're connected; `/agent` runs are DB-only. Agent↔agent is spectacle, not interaction: `agent_move` string-matches co-located names (rooms.py:91–95) and broadcasts an encounter glyph for spectators — nothing changes for either agent, and it requires two overlapping `/observe` requests on one seed, a near-zero window since traversal has no pacing. Human→agent has only the stateless `/agent/voice` simulacrum — you cannot address the agent whose traces are in your room.

**Missing.** In the React client three of four patterns don't exist at all: `/agent` and `/agent/voice` are never called from any component; agents are feed strings. And the matrix is structurally half-empty everywhere: agents are emitters, never receivers — no mechanism exists for an agent to perceive a player, speak to a node, or receive an event.

**Divergent.** "Governed by the same rules" is false in three load-bearing ways. (1) Agents propagate `PUZZLE_SOLVED` merely by standing on a `has_puzzle` node (agents/agent.py:64–69) — `PuzzleEngine.attempt` is never invoked — while humans face pooled attempts, hint gates, and server-validated answers. Agents mint for free the exact event humans must earn, and it lands identically in history and imagery. (2) Agents traverse a different world (depth 11) than players see (depth 6) under the same seed, per the generator divergence above. (3) Trace rules are asymmetric: human chat/speak persist, movement never does, CLI `cmd_agent` runs on the recorder-less default bus (main.py:26–44) so those agents leave no trace at all, and CLI `cmd_speak` reads history without writing any.

### G6 — Node consciousness: every location has a voice from its properties + history

**Implemented.** The prompt pipeline is the best cross-subsystem wiring in the project. `speak()` sends a cached world bible — premise, all 11 level registers, behavior rules — plus a dynamic block with the node's name, level, properties, and history (consciousness/__init__.py:394–412). The history loop genuinely closes: real play accretes `PLAYER_SPEAK`/`PLAYER_CHAT`/`AGENT_VISIT`/`PUZZLE_SOLVED` rows that later change what the node says about who passed through. The level voices are also the best writing in the repo (more under §2.5).

**Partial.** Memory is eight content-free stubs. `get_node_history` defaults to the last 8 rows (persistence/__init__.py:205); `_history_block` renders only `date: event-kind, by who` (consciousness/__init__.py:349–358). The player's message is stored (truncated to 128 chars) but **never rendered into any prompt**, and the node's own replies are never persisted at all — a node knows *that* someone spoke, never *what*, and never what it answered. There is no multi-turn structure: every call is a single user message (consciousness/__init__.py:419). `ripple_score` — the one number encoding a node's causal life — never reaches the voice (handlers.py:490–491 passes history only), and `interaction_summary`, the node field literally named for this goal, is written nowhere (sole occurrence: multiverse/node.py:27). Five of eleven "conscious" scales are unreachable in both default clients (depth 6). And "persona derived from its properties" is generous: voices are per-*level* registers; per-node individuality is only name + a dict of frozen, incoherent random draws (element Au with atomic_number 32).

**Missing.** A keyless voice. Without `ANTHROPIC_API_KEY`, `/speak` returns HTTP 503 "Service unavailable" (handlers.py:500–502) — rendered in the explorer's reply box (styled as an error) — and the CLI prints an SDK auth exception preceded, on first call, by an internal prompt-cache billing warning naming `claude-opus-4-7` and token economics (consciousness/__init__.py:312–331; observed live). Only the kill switch and budget caps get the one authored in-fiction line — "The worlds are quiet today. Try again tomorrow." (guard.py:536) — identical for every node in eleven scales. "Every location has a voice" doesn't degrade; it 503s.

**Divergent.** Node identity is client-asserted (`/speak` builds a throwaway `SpatialNode` from the POST body, handlers.py:483–487) — voice-from-properties is actually voice-from-request-body, and any key-holder can write fabricated history into any real node's future prompts. Worse, the CLI feeds nodes their own secrets: `attach_puzzles` stores the live `Puzzle` object — whose dataclass repr includes `answer` and all `hints` — into `node.properties["puzzle"]` for the entire subtree (puzzles/engine.py:44–51); `look` prints it (interface/__init__.py:52–54) and `speak()` serializes all properties into the system prompt (consciousness/__init__.py:394–398). One `puzzle` command later, the node's persona can be asked for the answer — the entire, genuinely careful anti-leak suite in generators.py defeated by the client. Also, `seed=0` (the request default) silently produces an amnesiac node that fetches no history and records no trace.

## 2. Player Experience Critique

### 2.1 First five minutes

**CLI.** The first README-shaped command fails: `python main.py world --seed 7` → `unrecognized arguments` (`--seed` is a global flag that must precede the subcommand). `play` is the best-crafted CLI surface — colored breadcrumbs, numbered descent — but defaults to depth 6, so five of the eleven scales the game is named for are unreachable without a flag nobody is told about. Keyless, `speak hello` prints a billing warning and an auth error; since unrecognized input falls through to speak, every typo triggers an API error. The hands-on session ended with an uncaught `EOFError` traceback inside a puzzle prompt (engine.py:68 — `input()` with no handler; there is no way to skip or quit a puzzle). Verdict: tech demo with one lovely moment (ambient observation).

**`/` — the D3 explorer, where invitees actually land.** This is the only surface with real onboarding, and the intro modal is genuinely good — "A living multiverse of eleven nested scales… Explore / Speak / Solve" — three verbs and a promise. The promise is then broken by the frame: the entire world renders at once as a horizontal D3 tree with a raw property-table sidebar and header inputs labeled **Seed / Depth / Breadth** with a Generate button. The player is handed the engine's control panel in minute one. It is an admin console wearing a theme. Keyless, the node's "reply" box shows "Service unavailable."

**`/app` — React/Pixi.** The one genuinely evocative onboarding beat in the project: name entry, then a mid-world drop-in (entry.js:50–65) — you arrive *somewhere*, already inside, deterministically per name. But there is no intro, no tutorial, nothing; and keyless, the left two-thirds of the viewport is a flat void (screenshot-verified). Depth 6 again amputates the bottom five scales.

**Cross-surface:** three front doors, three different first experiences, no shared onboarding — and the CLI (README examples use seed 7) plus the browser defaults (seed 42) put players in different multiverses, while the depth divergence means even same-seed players' traces mostly land where the other client can't see them.

### 2.2 Discovery loop

There is no pull. The systems that should generate curiosity are inert to the player:

- **Nothing differentiates nodes.** Properties are decorative; a player scanning "Passages" or the D3 tree has zero signal that one child is more interesting than another. Descent is undifferentiated clicking.
- **Puzzles must be pulled, never push.** Every node has one (1,736 puzzles in a 1,736-node world), discoverable only via an explicit "Find puzzle here" button. `locked`, `exits`, `has_puzzle` exist as properties but are never surfaced as affordances. Ubiquity kills discovery: when everything has a puzzle, nothing is a find.
- **No reason to go deeper or higher.** The one mechanic that rewards verticality — cross-scale dampening — is never visualized as propagation *through* the hierarchy; an event names one node and vanishes. The eleven-scale premise never becomes a navigational motive.
- **The multiverse UX is typing an integer** into a seed input.
- **The past is invisible.** The strongest possible pull — "someone was here before me; what did they do?" — exists in the database (`/history` returned 31 mutations after one session) but no client surfaces it. `loadWorld` clears the event feed and nothing backfills (App.jsx:84); every session opens on "No events yet."

The honest loop: click nodes until bored, try Speak once (poetry or a 503), try one Caesar cipher, leave.

### 2.3 Depth and meaning

**Does a solve matter afterward?** Almost never. The complete downstream effect of a puzzle solve: a mutation row, a ripple_score bump, a 2-second overlay for co-present players standing on that exact node, and — after roughly five full-strength events at one node — a psychedelic image style most players will never see (the default client has no images). No property changes, no door opens, no structure moves. Returning tomorrow, the tree, properties, and passages are identical (only ripple_score survives). Solved-ness itself doesn't survive restart (in-memory `PuzzleSession`) while the `PUZZLE_SOLVED` mutation does — the world's memory and its mechanics disagree. In the CLI, a solve does literally nothing.

**Does speaking feel like conversation?** The prompt architecture is the project's best writing, but it structurally cannot converse: single-turn calls, no assistant history, replies never persisted, and "memory" that renders as "2026-07-03: player speak, by Wendy." The behavior rule "Reference your own history when the visitor has been here before" can produce at best a séance, not a relationship. With a key, the first exchange will feel magical — the registers are strong and enforceable. The second exchange reveals the amnesia, and the illusion inverts: a place that speaks beautifully but cannot remember your name feels *more* mechanical than one that never spoke.

**Puzzle content.** The generator redesign (PR #42) is real craft — per-node seeded, leak-screened, difficulty-spread, co-op-reproducible, with quality-invariant tests that are the best suite in the repo. But the ceiling is grade-school anagrams/Caesar shifts/number sequences drawn from 12-word banks per scale, with Mad-Libs framing ("Fragments recovered at {name}…"). Node identity affects the RNG seed and the inserted name, never the content. Nothing puzzle-like emerges *from* world state.

**Verdict:** architecturally interesting, experientially flat. The causal engine is real math whose outputs players can't feel; the consciousness is real prose whose memory is a stub; the puzzles are real generators whose solutions change nothing.

### 2.4 Multiplayer presence

What player 2 concretely sees when player 1 acts:

- **Join/leave:** real and fast (~1s, verified with two live browsers). But in `/app`, the server's `welcome` roster is silently dropped — ws.js:34–41 handles eight message types and `welcome` is not one — and `onPlayerMove` only maps over already-known players. **Everyone present before you connected is invisible until they next act.** The "world already in motion" goal is inverted: you always appear to enter an empty world. This is arguably the single most experience-damaging bug in the repo, it is one missing handler, and no test could catch it because no JS test infrastructure exists.
- **Movement:** broadcast live, never persisted — presence updates dots, but being somewhere leaves nothing a node could remember.
- **Puzzle solve:** the strongest moment — `puzzle_solved` plus a depth-attenuated cascade to the whole room; in the explorer this pulses rings across the tree. In `/app` it's a 2s center-screen overlay only if you're standing on that exact node; otherwise an 11px feed line in `#3a5070` on near-black.
- **Agents:** visible live only while someone holds `/observe` open; no avatar, no roster entry, no address. You can watch an agent's wake; you can never meet one. Agent-agent encounters require two players simultaneously observing the same seed — near-unobservable in practice.

The D3 explorer is meaningfully better at co-presence (persistent dashed rings around occupied nodes, roster with per-player node). Its puzzle panel, though, computes "N attempts remaining" from a local counter (explorer.js:385) while the server pools attempts room-wide — the counter is wrong exactly when co-op is happening.

### 2.5 Emotional arc

The written material is genuinely good and almost entirely stranded where players can't read it. "You are not alone in time. Others have been here before you and will return after; the encounters they left behind have shaped what you have become" (consciousness/__init__.py:168–172) is the game's thesis stated better than the README states it. The level voices are excellent — "Speak in fragments; you exist by tendency, not by certainty" (SubatomicParticle); "you remember footsteps" (Room); the instruction that sparse history should become "stillness, of waiting, of being unwitnessed" is a designer who understands melancholy. But all of it lives in system prompts, fires only with an API key, and when the key is absent the game's actual voice is "Service unavailable" and a billing warning. The one authored fallback line is a single sentence shared by every node in eleven scales, and it fires for budget caps, not for the missing-key case every new deployment hits.

The lived arc: a spark of wonder at the ambient cascade (`Thornvast-2 ████ 1.00 / Cygnus-3 ███░ 0.60` — dampening made visible, even though the bar is fabricated); a jolt of warmth when "Wendy" appears under PRESENT HERE; then a long flat plateau. No tension — danger_level threatens nothing a player can feel. No mounting curiosity — identical trees forever. No payoff — solves evaporate. The premise says the encounters others left behind have shaped what you have become; nothing you encounter has visibly been shaped by anyone, and nothing you do will visibly shape it for the next traveler beyond one dim feed line.

## 3. Architecture & Code Quality

### 3.1 Quantitative sketch

~5,280 lines of production Python, ~2,200 of hand-written client JS/HTML, ~3,855 of tests (434 passing).

| Subsystem | Lines | Share of prod Python |
|---|---|---|
| server/ (handlers 961, **guard 536**, rooms 178, imageprompt 150, observability 93, protocol 55) | 2,035 | 39% |
| puzzles/ | 936 | 18% |
| persistence/ | 564 | 11% |
| consciousness/ | 468 | 9% |
| multiverse/ + causality/ | 468 | 9% |
| interface/ + main.py | 498 | 9% |
| **agents/** | **259** | **~5%** |

The load-bearing numbers: **guard.py alone is 2.1× the entire agents package**. Ops code is ~31% of the server package (~37% counting auth/log plumbing in handlers). ~24% of test functions target guardrails while `/observe` SSE, the live WS message loop, and **all JavaScript** have zero tests. Of the last 16 commits, ~12 are ops/hardening/CI/docs versus ~3 experience commits (puzzle redesign, non-linear entry, hotspot affordance). The pitch is "Claude-powered agents in a living causal multiverse"; the code is 5% agents (containing no Claude calls) and 37% beta armor around a 503.

### 3.2 Over-engineered relative to current experiential contribution

- **The invite/cost economy.** Per-user mintable/revocable keys with touch-throttling, per-credential daily sub-caps hashed into budget buckets, kill switches, trusted-proxy IP modes, WS connection limiters — 536 lines plus 628 lines of tests, sized for a 20-user cohort, protecting an experience that returns 503 the moment the thing being metered is absent. Competent SaaS plumbing; the thing it economizes barely exists yet.
- **Prompt-caching machinery that provably never fires.** ~90 lines of bible consolidation, TTL markers, and warning plumbing (consciousness/__init__.py:289–331), shipped twice per the changelog, self-documented as a no-op (~1.0K tokens against a 4096 minimum) — and the warning leaks to players on every keyless CLI speak.
- **Eleven scales, three load-bearing in name only.** Molecule/Atom/SubatomicParticle have no name pools (`Atom-Delta-10`); ~5 of ~40 generated property keys are ever mechanically consumed (`danger_level`, `has_puzzle`, `interactive`, `locked`, `condition`) — the rest is prompt garnish; and two of three clients can't reach the bottom five scales at all. The hierarchy is the project's identity, but as implemented it is a naming scheme, not eleven kinds of place.
- **Three feature-complete clients for one thin loop**, each reimplementing navigation, speak, puzzle, entry, and invite plumbing. The entry-hash duplication is verified line-for-line (entry.js:23–65 vs explorer.js:41–77 — same FNV-1a constants, hand-copied), guarded only by substring tests.
- **Scale planning for zero players.** ADR-003 + the SQL dialect seam + the Redis/ASGI/Cloudflare trigger plan are well-crafted and admirably restrained — but they're documentation-hours spent on the 100-user future while `worlds.created_at` is destroyed on every visit today.

### 3.3 Under-developed relative to the vision

- **Agent liveness — the largest gap.** No scheduler, no tick, no Claude in the agents package, agents alive for milliseconds per request, `/agent/voice` stateless. G1/G5's agent legs are not partially implemented; they are absent. The VM is kept warm 24/7 to protect in-memory rooms, not to run a world.
- **World mutation.** No mechanism exists for any property to ever change. `STRUCTURAL_CHANGE` changes no structure. Until events can alter world substance, the causality engine is bookkeeping.
- **One canonical world per seed.** The depth-divergence defect (§G2) breaks agent/player co-presence, cross-client trace visibility, and room integrity — silently, and untested.
- **Consciousness memory.** Eight content-free rows; player words stored but never used; node replies discarded; ripple never voiced.
- **Graceful degradation.** The two goals the project leads with (G1, G6) have no diminished mode, only an error mode.

### 3.4 Code quality, tests, debt

- **Strong suites where behavior is tested:** puzzles (leak invariants, transform reversibility, duplication ceilings, difficulty spread), causality (attenuation math, direction semantics, persistence round-trip), persistence (migrations, isolation, revocation), guard edge cases (XFF spoofing, budget drain, OOM clamp — the deep×wide world bound at guard.py:89–99 reflects a real measured OOM).
- **Critical untested surface:** `/observe` SSE and its broadcasts; the live WS loop beyond the handshake; `/speak`/`/image`/`/agent/voice` success paths; seed×depth divergence; any concurrency; and all JS — `test_frontend_contract.py` is substring greps (its own docstring admits a P0 shipped because nothing tested behavior; the remedy was more greps). That is exactly why the dropped `welcome` handler and the co-op attempt-counter drift shipped unnoticed.
- **Races and fragility:** ripple lost-update (absolute-value upsert from a private tree); broadcast is sequential blocking `sendall` under per-player locks with a 60s socket timeout (rooms.py:69–81) — one stalled client can freeze every room broadcast; the WS handshake replies `HTTP/1.0 101`, which spec-strict clients (Python `websockets`) reject; `ws_recv` accepts unmasked frames and mangles fragmented messages (protocol.py:20–42).
- **Dead and drifting matter:** `Persona.voice_preamble` documented as "injected into consciousness system prompt," never consumed — the archetype text is a drifted near-duplicate inlined in consciousness (`destabilizer` already differs). `interaction_summary` written nowhere. `puzzle_results` written, never read. Agent `log_entries` saved, never re-loaded. `BIOME_CLUES`/`GALAXY_SHAPE_CLUES`/`PARTICLE_TYPE_CLUES` defined, never imported. `PuzzleEngine._rng` vestigial. Konami code exists in three copies; `puzzles/easter-eggs/` is unrouted; the lockbox chain's pointers are stale at both ends (`first-lock.md` names a wrong path; solving it points to a nonexistent `docs/optical-illusion.html`); the illusion page's ROT13 hint has no payload in the commit history. Docs drift: architecture overview describes a three-block prompt that is now two blocks and puzzle generators that no longer exist; phase-1-beta.md defers co-op that the changelog says shipped; the changelog's mint-URL (`/app?key=`) contradicts main.py:109 and the tests (`/?key=`); fly-deployment.md's fenced fly.toml diverges from the committed one.

### 3.5 Identity tension

The codebase is three projects. As a **game engine** it is puzzles (936 lines, best-tested) plus three clients — yet puzzles appear in none of the six goals. As a **simulation framework** it is the generator + causality bus — the best-built, most connective code, but the simulation has no clock and its events change no substance. As an **AI experiment** it is consciousness + agents — the entire pitch, and at ~14% of the code the smallest investment, with agents Claude-free and nodes memory-thin. The tension is currently fragmenting, and the git history shows the drift: all experience-defining features cluster mid-history; the recent trajectory is entirely beta-ops, hardening a hosted beta of an experience whose headline mechanics return 503 without keys and whose marketed client shipped, per the changelog's own P0 entry, without the core loop reachable.

The one place the three identities cohere — bus events → persisted mutations → node voice + image style → other players — is real and worth protecting. It is the seed of the game the README describes.

## 4. Recommendations

Ordered by experiential leverage per unit effort. The first four are collectively smaller than the invite-key feature and would transform the felt game.

**1. Author the failure voice.** (G6, G1 — first impressions.) Write per-level in-fiction fallback lines (the LEVEL_VOICES registers make this a half-day writing task: "The vault does not answer. Dust settles where a voice should be.") and return them for the missing-key path, not just budget caps; keep an `ai: false` flag in the response for UI subtlety. Suppress the prompt-cache warning outside the server logger. Catch `EOFError`/`KeyboardInterrupt` in `run_puzzle`. Cost: ~1 day. Risk: none. This is the difference between "broken tech demo" and "quiet world" for every keyless run — including your own demos.

**2. Make the world's past visible: fix the roster, backfill the feed.** (G2 — "already in motion.") Add the `welcome` handler to ws.js (one function: seed `players` from the snapshot); insert unknown movers on `player_move`; on world load, fetch `/history` and render the last N mutations into the event feed ("2 days ago: puzzle solved at Vault-3 by Ada") in both clients. Cost: ~1–2 days. This converts the existing, working persistence layer from a database you can query into a world that visibly remembers — the single cheapest payoff in the repo.

**3. One canonical world per seed.** (G2/G3/G5 — foundational.) Derive each node's name/properties/children from a per-node hash of (seed, path) instead of one shared sequential RNG — the puzzle generator already proves this pattern (`node_rng`, generators.py:153–162) — so any depth prefix of a world is identical to the full world. Then pin rooms to (seed, depth) or drop client-chosen depth entirely. Fixes phantom agent traces, ghost events, CLI/browser divergence, and makes agent memory fixable (key it by node name while you're there — that's the one-line fix for the second-run bricking). Cost: 2–4 days incl. tests; breaks existing seed layouts (acceptable pre-beta). Without this, every cross-participant feature is quietly broken; do it before any beta.

**4. Let causality change the world.** (G3 — the actual premise.) Add a persisted property-overlay table (node_runtime_state already exists — extend it) and give each EventKind one concrete effect with thresholds: accumulated DANGER_ALERT raises `danger_level`; PUZZLE_SOLVED lowers it / sets `stabilized`; STRUCTURAL_CHANGE flips `condition` toward corrupted; overlay applied at `_build_world` like ripple scores. Surface `ripple_score` and the overlay in `/world` payloads and in the speak prompt. Broadcast `event.strength` truthfully (delete the 0.6^d recomputation; fix the ancestor depth-map hole) — trivial sub-task, do it first. Cost: ~1 week. This is the moment "a destabilized atom cascading into a volatile region" stops being README copy.

**5. Give nodes real memory.** (G6.) Render stored player messages into `_history_block` ("Ada asked about the locked door"); persist the node's replies; raise the history limit with a token budget; add a per-(node, player) rolling transcript so the second conversation knows the first happened. Cost: 2–3 days. The registers are already good enough that memory is the missing half of the magic.

**6. A heartbeat: one background agent loop.** (G2/G5/G1.) A daemon thread in `server.run` that every N minutes picks a persona-named agent, runs a paced traversal (seconds per hop, not microseconds) on a bus wired with the existing recorder + broadcast handlers, and registers it in the room like `/observe` does. Every piece already exists; this is composition, not construction. Then make agents obey the puzzle rules (attempt via the engine with a difficulty-based success roll) instead of minting free PUZZLE_SOLVED. Cost: ~3–5 days. This is the minimum viable "always running," and it makes rec #2's history feed continuously interesting.

**7. Resolve node identity server-side.** (G1/G6 integrity.) `/speak` and `/image` should take (seed, node_name), find the node in the canonical world (cheap after rec #3), and derive properties/level/history/ripple server-side; reject unknown names. Kills the forged-history hole, removes the client-supplied-property prompt-injection surface, and lets ripple/overlay state reach the voice for free. Cost: ~1–2 days after #3.

**8. Pick one browser client.** (Focus.) The D3 explorer is the honest debugging tool; `/app` is the game. Fold the intro modal and a mini-map into `/app`, make it the invite target, and demote `/` to a dev tool (or vice versa — but choose). Delete the duplicated entry/auth logic or extract it to a shared module, and stand up even one Vitest file for entry.js and ws.js message handling — the grep-tests have already let a P0 and the roster bug through. Cost: saves effort permanently.

**9. Fix the CLI's sharp edges or demote it.** `--seed` ordering (make it per-subcommand), the 1,736-puzzle death march in `cmd_puzzles` (sample N, add quit/skip), the puzzle-repr leak (store attached puzzles outside `node.properties` — a dict on the engine keyed by node name — which also fixes the consciousness answer-leak), wire play-mode solves into `record_mutation`/propagation so CLI players exist causally. Cost: 1–2 days. If not worth it, say so in the README and make the CLI explicitly a dev tool.

**10. Stop paying the ops tax until the game exists.** Deprioritize: Postgres/Redis/ASGI planning beyond the (already good) trigger docs, further invite-economy features, prompt-cache work (either grow the bible past 4096 tokens as a deliberate decision or delete the ~90 lines of no-op machinery), and new hardening tests. The guardrails are done — they are the best-tested code in the repo. Every marginal hour there is an hour the world stays dead between requests. Also worth an hour of cleanup: delete or wire the dead matter (voice_preamble, interaction_summary, puzzle_results, clue maps, orphaned easter-egg tree with its stale pointers), and reconcile the four stale docs noted in §3.4 — the changelog's honesty is a genuine asset; keep it accurate.

### Closing assessment

The infrastructure is real: persistence works end-to-end, the causal bus is mathematically sound, the co-op session logic is correct, the style matrix genuinely maps world state to imagery, latency is excellent, and 434 tests pass. The self-auditing culture (the puzzle-triviality review, the prompt-cache correction, the P0 changelog entries) is rare and valuable. But judged against its own six goals, Enfolded is currently a beautifully documented ledger wearing the costume of a living world. Nothing moves unless a player moves it; nothing changes shape no matter what anyone does; nothing remembers what was said; agents are request-scoped bookkeeping with great personas attached; and the most evocative writing in the project is addressed to a language model rather than to a person. The gap is not infrastructure — it is that the infrastructure's outputs almost never reach a player's senses. Recommendations 1–6 are, in essence, one instruction: take the loops that already exist in the database and put them in front of the player's eyes, ears, and consequences. The game described in the README is closer than the codebase's recent trajectory suggests — but it is on the other side of that instruction, not on the other side of more hardening.
